# [SQLD 단권화 #1] 데이터 모델링

> SQLD 1과목 1장 전반부 — 모델링의 개념, 3단계 모델링, 3단계 스키마, ERD

시험공부의 시작은 항상 "이게 왜 필요한지"를 아는 것이다. 데이터 모델링은 SQLD 1과목의 뼈대이고, 여기서 나오는 용어들이 뒤에 계속 재등장한다. 이번 편에서 큰 그림을 잡고 가자.

---

## 1. 모델링이란 무엇인가

**모델링(Modeling)** = 현실 세계의 복잡한 현상이나 사물을 **단순화해서 표현하는 기법**이다.

지하철 노선도를 떠올리면 된다. 실제 선로의 곡선, 거리, 지형은 다 생략하고 "어느 역 다음에 어느 역이 오는가"만 남긴다. 복잡한 현실에서 필요한 정보만 뽑아 약속된 방식(그림, 수식 등)으로 그리는 것, 그게 모델링이다.

### 모델링의 3가지 특징 — 암기: **단·추·명**

| 특징 | 의미 |
| --- | --- |
| **단순화** | 복잡한 현실을 기호·문자·그림으로 쉽게 표현 |
| **추상화** | 현실을 일정한 형식에 맞춰 간략하게 표현 |
| **명확화** | 애매모호함을 제거하고 명확하게 기술 |

### 정보시스템 모델링의 세 가지 관점

- **데이터 관점 (What)**: 업무가 *어떤 데이터*와 관련되는가
- **프로세스 관점 (How)**: 업무가 *실제로 무슨 일*을 처리하는가
- **상관 관점**: 업무 처리 방식에 따라 데이터가 어떤 영향을 주고받는가

---

## 2. 데이터 모델링의 3단계 — 개념 → 논리 → 물리

시험 단골. 각 단계의 **키워드**를 매칭하는 문제가 나온다.

### ① 개념적 데이터 모델링
키워드: **추상화, 전사적, ERD**

- 추상화 수준이 가장 높고, 업무 중심적·포괄적인 모델링
- **전사적 관점** — 전 부서가 공통으로 쓸 수 있는 통합 모델을 만든다
- 산출물은 주로 **ERD**

### ② 논리적 데이터 모델링
키워드: **키·속성·관계, 정규화, 재사용성**

- 시스템 구축을 위해 키, 속성, 관계를 정확하게 표현
- **정규화**를 통해 모델 품질을 높인다
- 재사용성이 높은 설계가 가능하다

### ③ 물리적 데이터 모델링
키워드: **성능, 인덱스, 반정규화**

- 실제 DB 구축을 위한 설계 단계
- 성능·저장 방식을 고려한 실제적 구현 방법 제시
- **인덱스, 역정규화(=반정규화)** 같은 기법이 여기서 등장한다

> 💡 역정규화: 정규화된 모델을 **다시(역) 합치거나 중복을 허용**해서 성능을 높이는 작업. 자세한 건 3편에서 다룬다.

## 3. 데이터 독립성 — 3단계 스키마 구조

DB를 세 개의 층으로 나눠서, **한 층이 바뀌어도 다른 층에 영향이 없도록** 만든 구조다.

```
[사용자]
   ↓
외부 스키마  ← 사용자 관점, 개인별 뷰(View)
   ↓ (논리적 독립성)
개념 스키마  ← 조직 전체의 통합 DB 구조
   ↓ (물리적 독립성)
내부 스키마  ← 실제 물리적 저장 구조
```

| 스키마 | 관점 | 설명 |
| --- | --- | --- |
| **외부 스키마** | 사용자 | 개인적인 DB 뷰, 응용 프로그램이 접근하는 영역 |
| **개념 스키마** | 통합 | 조직 전체의 DB 구조 정의 |
| **내부 스키마** | 물리 | 실제 데이터가 저장되는 방법 |

### 두 가지 독립성 (시험 포인트)

- **논리적 독립성**: 개념 스키마가 변해도 **외부 스키마에 영향 없음**
  - → DB 전체 구조를 수정해도, 사용자가 쓰는 화면(View)은 멀쩡해야 한다
  - 한 줄 암기: **논리적 독립성 = 사용자 뷰를 보호**
- **물리적 독립성**: 내부 스키마가 변해도 **개념 스키마에 영향 없음**
  - → 저장 장치를 바꿔도 DB의 전체 구조(ERD 등)는 그대로
  - 한 줄 암기: **물리적 독립성 = 저장 구조가 바뀌어도 구조는 유지**

층과 층 사이의 대응 관계는 **매핑(사상)**이라고 부른다.

## 4. 데이터 모델링의 3대 요소

| 요소 | 정의 | 예시 |
| --- | --- | --- |
| **엔터티(Entity)** | 정보를 저장하는 집합적 실체 | 학생, 주문 |
| **속성(Attribute)** | 엔터티를 구성하는 최소 데이터 단위 | 학번, 이름 |
| **관계(Relationship)** | 엔터티 간의 논리적 연관성 | 학생-수강-과목 |

각 요소의 자세한 내용은 다음 편(#2)에서 다룬다.

---

## 5. ERD (Entity-Relationship Diagram)

엔터티와 엔터티 간의 관계를 **그림으로 표현**한 것이다.

### ERD 작성 순서 (순서 문제로 출제됨)

1. 엔터티 도출
2. 엔터티 간 관계 정의
3. 관계명 기술
4. 관계의 참여도 기술
5. 관계 필수여부 기술

### ERD 표기법 3종

| 표기법 | 특징 |
| --- | --- |
| **Chen** | 가장 오래된 표기법. 관계를 **마름모**로 표시 |
| **IE** | 일명 **까마귀발 표기법**. 관계를 선으로 표시 |
| **Barker** | 엔터티에 **주식별자를 표기하지 않음**. 관계는 선 |

---

## 6. 좋은 데이터 모델의 조건

- **완전성**: 업무에 필요한 모든 데이터를 담는다
- **중복 배제**: 중복을 최소화해 일관성 유지
- **업무 규칙 반영**: 현업의 규칙을 정확히 표현
- **데이터 재사용**: 재사용이 쉬운 설계
- **의사소통**: 비즈니스 요구사항을 명확히 전달
- **통합성**: 전체 데이터의 통합된 시각 제공

## 7. 덤: 관계형 DB와 SQL 명령어 분류

이건 뒤 과목에서 계속 나오니까 지금 미리 눈에 익혀두자.

### DB의 발전

파일 시스템(60s) → 계층형/망형 DBMS(70s) → **관계형 DBMS**(80s, 오라클 등) → 객체 관계형(90s)

### SQL 명령어 4분류 (매우 자주 출제)

| 분류 | 뜻 | 명령어 |
| --- | --- | --- |
| **DML** | 데이터 조작 | SELECT, INSERT, UPDATE, DELETE |
| **DDL** | 데이터 정의 | CREATE, ALTER, DROP, TRUNCATE |
| **DCL** | 데이터 제어 | GRANT, REVOKE |
| **TCL** | 트랜잭션 제어 | COMMIT, ROLLBACK, SAVEPOINT |

### 테이블 구조 용어

- **컬럼(Column)** = 속성 = 필드
- **행(Row)** = 레코드 = 튜플
- **CHAR vs VARCHAR**: CHAR은 **고정길이**, VARCHAR는 **가변길이**

## 정리 한 장

1. 모델링 = 현실의 단순화. 특징은 **단순화·추상화·명확화(단추명)**
2. 데이터 모델링은 **개념(추상적·전사적·ERD) → 논리(키·정규화) → 물리(성능·인덱스·반정규화)** 순서
3. 3단계 스키마: 외부-개념-내부. **논리적 독립성은 뷰 보호, 물리적 독립성은 저장구조 변경 흡수**
4. 모델의 3대 요소는 **엔터티·속성·관계**
5. ERD 표기법: Chen(마름모), IE(까마귀발), Barker(주식별자 미표기)

다음 편에서는 3대 요소인 **엔터티·속성·관계·식별자**를 하나씩 뜯어본다.
