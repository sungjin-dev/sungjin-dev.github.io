# [SQLD 단권화 #3] 정규화와 반정규화

> SQLD 1과목 2장 — 정규화 단계(1NF~5NF), 이상 현상, 반정규화 기법

1과목에서 배점이 가장 큰 구간이다. 핵심 서사는 단순하다. **중복을 없애려고 테이블을 쪼갠다(정규화) → 조인이 늘어 조회가 느려진다 → 일부러 다시 합치거나 중복을 허용한다(반정규화).** 이 흐름만 잡으면 절반은 먹고 들어간다.

---

## 1. 정규화가 왜 필요한가 — 이상 현상(Anomaly)

**정규화(Normalization)** = 데이터의 일관성, 최소한의 중복, 최대한의 유연성을 확보하기 위해 테이블을 **분해**하는 과정. 근거는 **함수적 종속성(Functional Dependency)**이다.

정규화를 안 하면 생기는 3가지 이상 현상:

| 이상 현상 | 설명 |
| --- | --- |
| **삽입 이상** | 불필요한 데이터까지 같이 넣어야만 원하는 데이터를 추가할 수 있음 |
| **갱신 이상** | 중복 데이터 중 일부만 수정되어 데이터 불일치 발생 |
| **삭제 이상** | 필요한 데이터까지 같이 지워짐 |

---

## 2. 정규화 단계 — 예시로 한 방에 이해하기

### 제1정규형 (1NF): 모든 값은 원자값

한 칸에 값 하나만. 반복 그룹·다중값 속성을 제거한다.

```
Before: [학생: 김철수, 과목: 자바, 파이썬]   ← 한 칸에 두 개!

After (1NF):
  [학생: 김철수, 과목: 자바]
  [학생: 김철수, 과목: 파이썬]              ← 각 줄마다 값 1개씩
```

### 제2정규형 (2NF): 부분 함수 종속 제거

1NF를 만족하면서, 모든 비식별 속성이 **주식별자 전체**에 완전 종속되어야 한다. 복합 PK의 **일부에만** 종속되는 속성을 분리한다.

```
Before: PK가 [학번 + 과목]인데, '학생 이름'은 학번에만 의존 (과목과 무관)

After (2NF):
  [학생 테이블: 학번(PK), 이름]
  [수강 테이블: 학번(PK), 과목ID(PK)]       ← 이름을 따로 떼어냄
```

### 제3정규형 (3NF): 이행 함수 종속 제거

2NF를 만족하면서, **비식별 속성끼리의 종속**(A → B → C)을 제거한다.

```
Before: [학번(PK) → 부서번호 → 부서이름]

After (3NF):
  [학생 테이블: 학번(PK), 부서번호]
  [부서 테이블: 부서번호(PK), 부서이름]     ← 부서 정보를 따로 떼어냄
```

### BCNF: 모든 결정자는 후보키

3NF를 만족하면서, **후보키가 아닌 속성이 결정자인 경우**를 제거한다.

```
상황: [학생+과목]이 PK인데, '과목'만으로 '교수'가 결정됨
      (과목이 결정자인데 후보키가 아님)

After (BCNF): [과목-교수] 테이블을 별도로 분리
```

### 제4정규형 (4NF): 다치 종속 제거

서로 **독립적인 다중값**들이 한 테이블에 있으면 조합이 곱해져서 데이터가 뻥튀기된다.

```
Before: (스킬 × 프로젝트 모든 조합이 저장됨)
  [김철수, 자바, 쇼핑몰]
  [김철수, 자바, 메신저]
  [김철수, 파이썬, 쇼핑몰]
  [김철수, 파이썬, 메신저]   ← 무의미한 중복

After (4NF): 상관없는 다중값을 테이블 2개로 완전 분리
  [개발자-스킬: (김철수, 자바), (김철수, 파이썬)]
  [개발자-프로젝트: (김철수, 쇼핑몰), (김철수, 메신저)]
```

### 제5정규형 (5NF): 조인 종속 제거

분해한 테이블들을 다시 조인했을 때 **원래 테이블과 동일**해야 한다.

### 단계별 암기 요약

> **1NF(원자값) → 2NF(부분종속 제거) → 3NF(이행종속 제거) → BCNF(결정자=후보키) → 4NF(다치종속 제거) → 5NF(조인종속 제거)**

## 3. 정규화와 성능

정규화는 조회 성능을 **떨어뜨릴 수도, 올릴 수도** 있다. (단정형 선지 주의!)

- 테이블이 쪼개지니 **조인이 늘어 조회가 느려질 수 있다**
- 반면 중복이 제거되어 **특정 조회는 오히려 빨라진다**
- **입력/수정/삭제 성능은 일반적으로 향상**된다 (중복 데이터가 줄어서)

---

## 4. 반정규화 (Denormalization)

### 가장 중요한 한 문장

> **반정규화는 "정규화를 안 하는 것"이 아니라, "정규화된 것을 일부러 다시 무너뜨리는 것"이다.**

순서가 핵심: **정규화 → (조회 성능 저하 발생) → 반정규화 적용**

### 반정규화는 최후의 수단

성능 문제가 생겼다고 바로 반정규화부터 하면 안 된다. **구조를 바꾸지 않는 방법을 먼저 시도**한다. (시험 포인트)

반정규화 전에 검토할 것:
1. **뷰(View)** 활용
2. **인덱스** 조정
3. **클러스터링** 적용
4. **파티셔닝·캐시** 적용

이걸로도 안 될 때 비로소 반정규화한다.

### 정규화 vs 반정규화 트레이드오프

| 구분 | 정규화 | 반정규화 |
| --- | --- | --- |
| 목적 | 데이터 무결성, 중복 제거 | **조회 성능 향상** |
| 단점 | 조인 증가로 성능 저하 가능 | 중복으로 **무결성 깨질 위험** |
| 유리한 작업 | 입력/수정/삭제(DML) | 조회(SELECT) |

> 반정규화의 대가: 데이터가 중복 저장되니 **입력·수정·삭제 시 여러 곳을 동시에 고쳐야 하고**, 안 그러면 불일치가 발생한다. 이게 가장 큰 리스크다.

---

## 5. 반정규화 기법 — 대상별 정리

| 대상 | 기법 |
| --- | --- |
| **테이블** | 병합 / 분할(수직·수평) / 추가(중복·통계·이력·부분) |
| **컬럼** | 중복 컬럼 / 파생 컬럼 / PK에 의한 컬럼 추가 등 |
| **관계** | 중복 관계 추가 |

### 테이블 반정규화

- **병합**: 1:1 또는 1:M 관계 테이블을 하나로 합침
- **분할**
  - **수직 분할** = **컬럼**을 나눔 (자주 쓰는 컬럼 / 안 쓰는 컬럼 분리)
  - **수평 분할** = **행**을 나눔 (지역별, 연도별 등)
- **추가**: 중복 테이블(서버가 다를 때), 통계 테이블(미리 계산), 이력 테이블, 부분 테이블(자주 쓰는 컬럼만)

>  수직/수평 헷갈리면: "수직은 세로선 = 컬럼을 가름", "수평은 가로선 = 행을 가름"

### 컬럼 반정규화

- **중복 컬럼 추가**: **다른 테이블에 이미 있는 값**을 복사 (조인 줄이려고)
- **파생 컬럼 추가**: **계산 결과값**을 저장 (합계·평균, 매번 계산하기 아까울 때)
- 이력 테이블 컬럼 추가, PK에 의한 컬럼 추가 등

>  중복 컬럼 = "복사해온 값", 파생 컬럼 = "계산해서 만든 값". 구분 문제로 나온다.

### 관계 반정규화

- **중복 관계 추가**: 조인 경로를 단축하려고 이미 있는 관계를 하나 더 추가

---

## 정리 한 장

1. 정규화는 **이상 현상(삽입·갱신·삭제)** 제거가 목적이며, 함수적 종속성이 근거다
2. 단계: **1NF 원자값 → 2NF 부분종속 제거 → 3NF 이행종속 제거 → BCNF 결정자=후보키**
3. 정규화는 DML 성능↑, 조회는 조인 증가로 느려질 **수도** 있다 (단정 금지)
4. 반정규화는 **정규화 이후, 뷰·인덱스 등 다른 수단을 먼저 검토한 뒤** 쓰는 최후의 수단
5. 수직분할=컬럼, 수평분할=행 / 중복컬럼=복사값, 파생컬럼=계산값
6. 반정규화의 대가는 **무결성 위험** — 항상 세트로 언급된다

다음 편은 **트랜잭션(ACID)과 NULL**, 그리고 본질/인조 식별자와 계층형 모델이다.
