C언어에서 가장 당황스러운 순간 중 하나가 바로 이 name = "SungJin";이 안 된다는거다. 

그 이유를 메모리 관점에서 살펴보자. 

## 1. 왜 name = "SungJin"은 안 될까?

C언어에서 배열(문자열)은 선언될 때 '딱 정해진 메모리 주소'를 할당받는다.

```c
#include <stdio.h>
#include <string.h>

char name[20]; // 메모리에 "name"이라는 이름의 20바이트 공간이 확보됨

name = "SungJin";  // 에러 발생
```

여기서 name은 20바이트 공간의 **'시작 주소'**를 가리키는 `상수`이기 때문에, 그 주소를 가리키는 고정된 이름을 다른 주소를 가리키지 못하게 막는 것이다. 

 이런 경에 String Copy(문자열 복사)를 이용하면 된다. 

## 2. strcpy 사용법

`strcpy`는 주소를 바꾸는 게 아니라, 해당 메모리 주소로 찾아가서 안에 있는 데이터를 덮어쓰라는 명령이다.

strcpy(name, "SungJin") : "현재 주소로 (SungJin)을 들고 가서, 너를 채워 넣어라" (성공!)


## 3. C언어의 낭만(?) - 깐깐함의 이유

파이썬이나 자바스크립트는 name = "SungJin"이라고 하면, 내부적으로 메모리 주소를 갈아치우는 작업을 엔진이 알아서 해준다. 

하지만 C언어는 "메모리 주소는 프로그래머가 관리해야 한다"는 철학 때문에, 프로그래머가 직접 `strcpy`라는 복사 명령을 내려야만 비로소 메모리 내용이 바뀐다. 이렇게 함으로써 데이터가 어디서 어디로 복사되는지를 프로그래머가 완전히 통제할 수 있게 된다.

### 1) C언어의 메모리 접근 방식
C언어에서는 "문자열 변수에 값을 직접 대입(=)할 수 없기 때문에" strcpy 함수를 사용한다. 이는 단순히 값을 복사하는 것을 넘어, "메모리 주소로 직접 찾아가 데이터를 덮어쓰는 과정"이다.

### 2) 메모리 관리: 장인 VS 로봇청소기
이러한 메모리 관리 방식을 청소에 비유하자면 다음과 같다.

`가비지 컬렉터(GC)`: "알아서 구석구석 쓰레기 메모리를 치워주는 로봇청소기"

`C/C++ 프로그래머`: "집 구조와 먼지가 쌓이는 곳을 완벽히 파악해 직접 빗자루질을 하는 장인"

Java, Python, C#, JavaScript 등 대다수의 현대 프로그래밍 언어는 가비지 컬렉터를 지원하여, 더 이상 참조되지 않는 메모리를 알아서 정리해 버린다. 반면, C/C++에서는 개발자가 메모리를 직접 해제해야 하며, 이를 깜빡할 경우 `메모리 누수(Memory Leak)`라는 치명적인 버그에 봉착하게 된다.

### 3) 메모리 관리 패러다임의 진화
결국 프로그래밍 언어의 메모리 관리는 "성능을 위해 직접 치울 것인가(C, C++)", "편리함을 위해 로봇에게 맡길 것인가(Java, Python)", 아니면 "새로운 규칙으로 방을 깔끔하게 유지할 것인가(Swift, Rust)"의 세 가지 방향으로 진화하고 있다.

새로운 규칙을 도입한 대표적인 예시는 다음과 같다.

**Swift, Objective-C** (`ARC 방식` - 스마트 휴지통)
로봇청소기가 온 집안을 돌아다니는 대신, 데이터에 '현재 사용 중인 참조 횟수'를 꼬리표처럼 적어둡니다. 이 횟수가 0이 되는 순간, 즉각적으로 그 자리에서 메모리를 치워버리는 효율적인 방식입니다.

**Rust** (`소유권 방식` - 미니멀리스트)
요즘 가장 주목받는 C/C++의 대체 언어. 아예 코드를 컴파일(번역)하는 단계에서 "누가 언제 메모리를 버릴지"에 대한 규칙을 완벽하게 증명해 낸다. 쓰레기가 방치될 가능성 자체를 문법적으로 원천 차단해 버리는 매우 깐깐하고 안전한 방식이다.
