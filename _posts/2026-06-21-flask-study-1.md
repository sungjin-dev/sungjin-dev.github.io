우선 플라스크 웹서버를 구현할 때 아주 중요한 도구인 `render_template`라는 아이에 대해 알아보자.
<br><br>
내가 생각하는 가장 중요한 함수다. 
<br><br>
물론 진짜 실질적인 1등 공신은 @app.route()! 
<br><br>
플라스크라는 프레임워크의 정체성 그 자체는 사실 "라우팅(URL과 함수를 연결)"이라고 봐도 무방하기 때문이다. 
사실 `render_template`은 화면을 예쁘게 그려주는 역할일 뿐, 길을 뚫어주는 `route`가 없다면 웹 서버 자체가 성립하지 않는다.
<br><br>
하지만 배우는 입장에서 `render_template`을 잘 활용하는게 더 중요하다. 
<br>
먼저 저 함수들을 이해하기 위해 우리는 템플릿 엔진 개념에 대해 알아야한다. 
<br><br>
 
## 1.템플릿 엔진(Jinja2)이란?
<br>
 HTML이라는 **'빈 틀'**에 **데이터**라는 '내용물'을 규칙에 따라 찍어내 웹페이지를 만들어내는 방식을 말한다.   
<br>
우리가 보는 웹 서버가 매번 똑같은 HTML만 보여주는 게 아닌, 상황에 따라 가변적으로 보여줄 수 있도록 도와주는 아주 고마운 기술이다.
<br>
* **데이터:** 앱(App)에서 처리된 변수나 객체 정보
* **템플릿:** 데이터가 들어갈 자리가 표시된 HTML 파일 (예: {% raw %}{{ name }}{% endraw %})
* **렌더링(Rendering):** 두 요소를 합쳐 브라우저가 읽을 수 있는 HTML로 변환하는 과정

## 2. 작동 원리

1. **앱(App) 측:** `render_template` 함수를 호출하며 템플릿 파일 이름과 데이터를 인수로 전달합니다.
2. **템플릿 엔진(Jinja2) 측:** HTML 파일을 단순 문서가 아닌 '데이터 치환이 필요한 문자열'로 인식합니다.
3. **치환 과정:** HTML 내의 {% raw %}{{ 변수명 }}{% endraw %} 문법을 실제 데이터로 모두 치환합니다.
4. **전달:** 완성된 HTML 문서가 사용자 브라우저로 전송됩니다.
<br><br>
##3. 주요 특징
<br>
**유연성**: HTML의 거의 모든 위치에서 변수를 마음대로 삽입할 수 있으며 
<br>
**동적 생성**: 사용자마다 다른 정보를 실시간으로 HTML에 반영할 수 있어, 게시판, 프로필 페이지 등을 만들 때 필수적이다. 
<br><br>

**자료.1**
```html
# app.py

from flask import render_template

@app.route('/')
def index():
               #  'psj'라는 데이터를 템플릿 엔진에 전달
    return render_template('index.html', name='psj')
```
<br>

**자료.2**
{% raw %}
```html
<body>
    <h1>Name : {{ name }}</h1>
    <button onclick="greet('{{ name }}')">안녕</button>
</body>
```
{% endraw %}

<br>
h 태그의 `{% raw %}Name : {{name}}{% endraw %}`는 템플릿 엔진이 가장 흔하게 사용하는 방식이며, 
<br><br>
데이터가 name = 'psj'라면 최종적으로 <h1>Name : psj</h1>가 되어 브라우저에 표시되게 된다. 
<br><br>
* `{% raw %}{{ 변수명 }}{% endraw %}` : 변수의 값을 그 자리에 그대로 출력 
<br><br>
* `{% raw %}{% 제어문 %}{% endraw %}` : if 조건문이나 for 반복문 등 로직을 제어할 때 사용. 화면에 글자로 출력되지 않고 구조를 결정한다고 보면 되는데 
<br><br>
주로 if문은 말 그대로 조건에 따라 결과값을 제어하는 형태고, 
<br>
for문은 주로 리스트나 딕셔너리 같은 컨테이너형 자료를 넘겨 받아 작업하는 경우가 흔하다. 
<br>
* `{% raw %}{% include %}{% endraw %}` : 제어 명령문으로 중복되는 코드들을 다른 파일에 몰아 넣고 {% raw %}{% include 'footer/custom.html' %}{% endraw %}만 써놓으면 알아서 해당 중복 태그들을 인식한다.  
<br><br>
*`{% raw %}{# 주석 #}{% endraw %}` : 템플릿 전용 주석으로, 브라우저의 '소스 보기'를 해도 나타나지 않는다. 참고로 알아 두자. 
<br><br>
위의 자료 2번을 보면 button 태그 안에 {% raw %}onclick="greet('{{ name }}')"{% endraw %}을 넣어 버튼을 클릭했을 때 자바스크립트를 실행시켜 다양한 효과를 줄 수 있다. 
<br><br>
여기서 `on`이라는 접두사가 붙으면 자동적으로 자바스크립트라고 인식하게 되며 해당 함수가 실행되는데 
<br><br>
`{% raw %}{{ name }}{% endraw %}`을 인수로 받아서 자바스크립트 내 코딩으로 `{% raw %}{{ name }}{% endraw %}`그대로 받아와서 사용이 가능하다. 
<br><br>
그런데 개발자 입장에서는 코드가 눈에 보이지만 실제로는 jinja2말고는 {% raw %}{{ name }}{% endraw %} 이런 이중 괄호를 해석할 수가 없다. 그런데 어떻게 로직이 돌아갈까? 
<br><br>
예를 들어 파이선 서버의 `render_template` 함수가 `user_id`(실제로는 sungjin)를 html에 넘기고, html문서 `{% raw %}{{user_id}}{% endraw %}`와 같은 이중괄호 안 인자들을 템플릿 엔진이 전부 해석해서 치환할 것이다. 
<br><br>
그럼 실제 값인 `sungjin` 그대로 `onclick함수`의 인자로 들어가서 자바스크립트 함수가 실행된다. 자바스크립트 내 코딩에서는 그대로 `{% raw %}{{user_id}}{% endraw %}` 이렇게 써서 로직을 돌리지만 실제로 데이터가 옮겨 가는 과정을 보면 이미 실제 값으로 변환된 이후에 넘어간다는 사실을 알아 두자. 
<br><br>
## 4. render_plate가 중요한 이유
<br>
자바스크립트 코드는 사용자의 브라우저에서 실행되기 때문에, **'F12'**를 누르면 누구나 쉽게 코드와 데이터를 전부 들여다볼 수 있다.
<br>
만약 자바스크립트로 회원 정보나 로그인 로직을 제어한다면 DB에서 데이터나 비밀번호 검증 로직이 사용자 브라우저로 최소 한번은 넘어 올 때 보안에 굉장히 취약하다. 
<br>
반면 `Flask(render_template)`를 쓰면, 데이터 조회와 비밀번호 검증 등의 핵심 작업은 안전한 파이썬 서버에서 모두 끝내고, 
<br>
그 결과물인 "로그인 성공!"이라는 **껍데기(HTML)**만 브라우저로 보내기 때문에 데이터가 절대 유출될 일이 없다. 
<br><br>
참고로 이 소중한 함수 사용법은 간단하다.
<br>
```
python

#app.py

from flask import Flask, render_template 
                    #플라스크 모듈에서 꺼내온다
 
@app.route('/name/<name>')   
def show_name(name):
		       
	 return render_template('index.html', name=name)
   변수를 해당 파일경로의 html 템플릿 엔진에게 건넨다. 
   
```
<br><br>

## 5. render_template vs redirect

<br>
`render_template`과 `redirect`는 웹 개발을 할 때 혼동될 수 있는데 둘의 차이는 **"누가 화면을 결정하는가?"**를 보면 된다. 
<br><br>
1) `render_template` (서버가 **직접** 화면을 생성)
<br>
파이썬 서버가 준비된 HTML 템플릿을 찾아서 **render_template**을 통해 데이터를 채워 넣은 뒤 그 결과물을 브라우저에게 그대로 보여준다.
즉 쉽게 말해 html문서를 템플릿 엔진이 렌더링해서 화면에 직접 그려낸다고 보면 된다. 
<br>
point : 특히 화면에 넘겨줄 데이터(변수)가 복잡할 때 주로 사용한다. 과부하나 메모리 문제만 아니면 이론상으로는 무한대도 가능할지도...
<br>
2) `redirect` (브라우저에게 새로운 주소를 명령)
<br>
서버가 브라우저에게 "지금 보고 있는 페이지 말고, 다른 주소로 다시 접속해!"라고 명령 
<br>
"로그인 성공했으니까 메인 페이지(대시보드)로 보내버리자."
<br>
point : redirect를 하면 이전 페이지에서 쓰던 변수들은 전부 증발하고, 새로 접속하는 주소의 함수가 다시 실행된다. 
<br><br>
## 6. 그밖의 함수들 (ur_for)
<br>
Flask와 같은 웹 프레임워크를 공부하다 보면, 단순히 URL을 직접 타이핑하지 않고 **url_for**라는 함수를 사용하는 것을 볼 수 있다.
<br><br>
그럼 왜 굳이 번거롭게 **url_for**를 사용하는지, 이것이 왜 **"유지보수 가능한 코드"**의 핵심인지 자세히 알아보자. 
<br><br>
1) **왜 URL을 직접 쓰면 나중에 고생하는가?** (하드코딩의 위험성)
<br><br>
우선  **url_for**이란 "라우트 함수 이름으로부터 URL을 역으로 생성해 주는 함수다. Mapping을 통해 대응하는 값을 찾아낸다고 봐도 된다.   
<br><br>
순방향: URL -> 함수 (서버가 하는 일)
<br><br>
역방향(url_for): 함수 -> URL 
<br><br>
보통  HTML 코드나 서버 코드에서 특정 페이지로 이동하기 위해 URL을 직접 작성한다. 
<br><br>
예를 들어, 로그인 페이지로 이동하는 링크를 만든다면 다음과 같을 것이다.

<br><br>
```
<a href="/login">로그인 고고씽!</a>
```
<br><br>

하지만, 이 방식에는 치명적인 단점이 있다. 만약 /login이라는 주소를 /user/login으로 바꾸고 싶다면 프로젝트 내의 모든 HTML 파일과 파이썬 코드를 뒤져서 일일이 수정해야한다.
<br><br>
굉장히 번거러울 뿐 아니라 개발자도 인간이기에 실수를 할 확률도 존재한다. 
<br><br>
2)** 왜 url_for를 써야 하는가?** 
<br><br>
`url_for`를 사용하면 위와 같은 하드코딩(Hard-coding) 문제를 깔끔하게 해결할 수 있다. 
<br><br>

{% raw %}
```
<a href="{{ url_for('login_page') }}">로그인 고고씽!</a>
```
{% endraw %}
<br><br>
여기서 'login_page'는 실제 URL 주소가 아니라, 파이썬 코드(app.py)에서 정의한 함수 이름이다. 
<br><br>
```
@app.route('/login') # 주소는 마음껏 바꿔도 됨
def login_page():    # 함수 이름은 반드시 고정해야 마음대로 서버 주소를 바꿀 수 있다
    return render_template('login.html')
```
<br><br>
**유지보수의 편의성**: 서버에서 @app.route의 주소 경로를 아무리 바꿔도, `url_for`는 여전히 해당 함수 이름(login_page)을 찾아가기 때문에 HTML 코드를 한 줄도 수정할 필요가 없다.
<br><br>
이것이 **결합도(Coupling)**를 낮추는 방법이자 데이터 무결성 원칙을 수호하는 방식이기도 하다. 
<br><br>
결론적으로 `url_for`를 사용하여 '주소'가 아닌 **함수 이름**을 참조하면 코드를 변경할 때 수정 범위가 획기적으로 줄어든다.
<br><br>
좀 더 개발자스러운 접근을 해보면 
<br><br>
이러한 모든 과정이 결국에는 결합도를 낮추고 **데이터 무결성**을 유지하기 위한 방법임을 알 수 있다. 
<br><br>
## 7. **결합도(Coupling)**와 **데이터 무결성(Data Integrity)**과의 관계
<br><br>
여기서 데이터 무결성은 '데이터가 정확하고 일관성 있게 유지되는 것'을 의미한다. 
<br><br>
결합도가 높은 상황(High Risk)을 생각해보자 
<br><br>
경제학에서도 **커플링(Coupling)** **디커플링(Decoupling)**이라는 용어를 사용하는데,
<br><br>
예를 들어 미국 중국 한국 일본 등등 각국의 경제는 굉장히 밀접하게 연관되어 있다. 
<br><br>
물론 장점도 있겠지만 어떤 한 경제가 무너지거나 금융 외환위기(2008 리먼브라더스 사태 등) 등이 터졌을 때,
<br><br>
아무 문제 없던 주변 나라들 조차 도미노 처럼 우르르 위기를 맞게 된다. 
<br><br>
코딩도 마찬가지다. 여러 함수나 모듈들을 따로 분리하지 않고 한 페이지에 전부 하드코딩해놓았다고 해보자. 
<br><br>
A라는 함수나 모듈이 데이터를 수정하면서 데이터를 초기화하거나 잘못저장해버리면, B라는 모듈은 깨지거나 이미 사라져버린 데이터를 받게 될 것이다. 
<br><br>
하지만 따로 따로 모듈화(객체화)해서 관리한다면 서로 독립적으로 운영되기 때문에 설사 실수를 하거나 에러가 나도 영향도 적을 뿐 아니라 그 부분만 보수하면 된다. 
<br><br>
물론 "객체화(Objectification)보다는 **인스턴스화(Instantiation)**라는 용어를 사용하는 것이 프로그래밍적으로 더 정교한 표현이다. 둘다 알아두자!
<br><br>
결국 이렇게 결합도를 낮추는 것이 궁극적으로 데이터 무결성을 유지하는 방법이다.
