# 자바 코드는 어떻게 실행되는가 

`System.out.println("Hello")` 한 줄이 화면에 글자로 출력되기까지 생각보다 긴 여정이 숨겨져 있다. <br>
컴파일, 클래스 로딩, 상수풀 해석, 그리고 기계어 번역 등등 이런 구조적인 흐름을 파악하고 코딩을 하는 것은 생각보다 굉장히 큰 도움이 된다. 
작자는 부끄럽지만 이 구조적인 흐름을 이해하는데 꽤나 오랜시간을 소요했다. 
그래서 이 부분에 대해 누군가는 쉽게 이해하길 바라는 마음에서 글을 작성해보았다.  


간단하게 나타내면 다음과 같다.

```
자바 코드 (.java)
   │
   │  ① javac 컴파일 (개발자 컴퓨터에서 1회)
   ▼
바이트코드 + 상수풀 (.class)
   │
   │  ② 클래스 로더가 메타스페이스에 로드 (실행 시)
   ▼
메타스페이스의 클래스 구조 정보
   │
   │  ③ 실행 중 #번호를 상수풀에서 해석(resolve)
   ▼
   │  ④ 인터프리터 실행 + 자주 반복 실행되는 코드는 JIT가 기계어로 번역
   ▼
CPU 기계어 실행
```

## 1. 왜 곧바로 기계어로 컴파일하지 않는가

C 언어는 컴파일하면 곧바로 기계어가 나온다. 그런데 기계어에는 치명적인 특성이 있다. **CPU 종류마다 언어가 다르다**는 것이다. 인텔 x86용으로 컴파일한 프로그램은 ARM 칩에서 돌아가지 않는다. 운영체제·CPU 조합마다 따로 컴파일해서 배포해야 한다.

자바는 이 문제를 **중간 언어**라는 발상으로 풀었다. 실제 CPU가 아니라, **JVM이라는 가상의 기계를 위한 언어(바이트코드)**로 먼저 컴파일해 두는 것이다.

```
[C 방식]   소스 ──컴파일──> 인텔용 기계어, ARM용 기계어, ... (기계마다 따로)

[자바 방식] 소스 ──컴파일──> 바이트코드 (딱 하나)
                              │
              윈도우 JVM ─────┼───── 맥 JVM ───── 리눅스 JVM
              (각자 자기 기계의 언어로 번역)
```

바이트코드는 어떤 CPU의 언어도 아니므로, 어떤 기계에도 종속되지 않는다. 번역의 부담은 각 기계에 설치된 JVM이 진다. 윈도우용 JVM, 맥용 JVM이 따로 배포되는 이유이고, 자바의 간판 슬로건 **"Write Once, Run Anywhere(한 번 작성하면 어디서든 실행)"**의 정체가 바로 이 구조다.

여기서 오해 하나를 정리하자. 바이트코드도 파일로는 이진 데이터지만, **CPU가 직접 읽는 기계어가 아니다.** "이진수 = 기계어"가 아니라, 기계어가 되려면 아직 한 번의 번역(④단계)이 남아 있다.

## 2. javac — 본문과 색인을 함께 만드는 컴파일러

`javac`(Java Compiler)는 .java 파일을 .class 파일로 컴파일한다. 그런데 이 .class 파일의 내부 구조가 재미있다. 한 권의 책처럼 **두 부분**으로 되어 있다.

- **바이트코드** — 실행 명령이 줄줄이 적힌 **본문**
- **상수풀(constant pool)** — 본문이 참조하는 이름·값들의 **색인**

왜 이렇게 나눴을까. 바이트코드 본문은 모든 것을 **번호로만** 지시하기 때문이다. 실제 코드로 보자.

```java
public class Member {
    String name = "SungJin";

    void writePost() {
        System.out.println(name);
    }
}
```

이걸 컴파일하면 상수풀에는 대략 이런 항목들이 번호를 받아 등록된다.

| 번호(예시) | 종류 | 내용 |
|---|---|---|
| #2 | 클래스 참조 | Member |
| #5 | 필드 참조 | Member의 name (String 타입) |
| #7 | 문자열 리터럴 | "SungJin" |
| #9 | 클래스 참조 | java.lang.System |
| #11 | 필드 참조 | System의 out |
| #12 | 메서드 참조 | println(String) |

그리고 본문(바이트코드)에는 이렇게만 적힌다.

```
ldc           #7    // "7번 항목을 로드하라"   → "SungJin"
getfield      #5    // "5번 필드를 가져와라"   → name
invokevirtual #12   // "12번을 호출하라"       → println
```

즉 **클래스 이름, 필드, 메서드, 문자열 리터럴처럼 본문에 직접 쓰기엔 긴 것들이 전부 번호표를 받고 색인으로 빠진다.** 본문은 짧은 번호로만 지시하니 파일이 작아지고, 같은 이름을 백 번 써도 색인에는 한 번만 등록하면 된다.

이건 상상도가 아니라 직접 확인할 수 있다. 터미널에서 아래 명령을 치면 실제 상수풀 목록과 바이트코드가 그대로 출력된다. 구조 이해에 이만한 실습이 없으니 꼭 한번 해보자.

```bash
javac Member.java
javap -v Member.class
```

## 3. 클래스 로더 — 설계도를 메타스페이스에 올린다

`java` 명령으로 프로그램을 실행하면, **클래스 로더**가 필요한 .class 파일을 읽어 메타스페이스에 올린다. 이때 올라가는 것은 상수풀만이 아니라 **클래스 구조 전체가 한 세트**다.

```
┌─ 메타스페이스 ──────────────────────────────┐
│  ┌─ Member 클래스 구조 정보 ─────────────┐  │
│  │  · 바이트코드 (명령부 = 본문)          │  │
│  │  · 런타임 상수풀 (색인부)              │  │
│  │  · 필드/메서드/생성자 정보             │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

본문과 색인은 책 한 권의 앞뒤 페이지 같은 관계이므로 같은 서가에 꽂히는 게 당연하다. .class 파일의 상수풀이 메모리에 로드된 것을 **런타임 상수풀**이라고 부르며, 클래스 파일마다 자기 색인을 갖고 태어나므로 런타임 상수풀도 **클래스마다 하나씩** 존재한다.

여기서 "상수풀"이라는 이름에 대한 오해도 짚고 가자. 이름 때문에 `final` 상수들을 모아둔 창고 같지만, 실체는 **컴파일 시점에 확정된 이름표와 리터럴의 색인표**다. 불변이라서 메타스페이스로 보낸 게 아니라, **설계도의 일부라서 설계도와 함께 있는 것**이다(설계도답게 내용이 불변일 뿐이다). 참고로 문자열 리터럴의 **실제 String 객체**는 힙의 String Pool에 생기고, 상수풀에는 그것을 가리키는 정보만 남는다 — 지난 글에서 "문자열 풀은 힙으로, 런타임 상수풀은 메타스페이스에"라고 갈라 말했던 이유다.

클래스 로딩의 한 가지 특징: **필요할 때 올린다.** 프로그램 시작 시 모든 클래스를 다 로드하는 게 아니라, 실제로 처음 사용되는 시점에 로드한다. 안 쓰이는 클래스는 끝까지 로드조차 되지 않을 수 있다.

## 4. 해석(resolve) — 번호의 정체를 실행 중에 푼다

JVM이 바이트코드를 실행하다가 `invokevirtual #12`를 만나면, 그 클래스의 런타임 상수풀에서 12번 항목을 찾는다. 거기엔 "java.io.PrintStream의 println 메서드, String을 받고 void를 리턴"이라는 **이름표(심볼릭 참조)**가 적혀 있다. JVM은 이 이름표를 실제 메모리 주소로 바꿔치기하는데, 이 과정을 **해석(resolution)**이라고 한다.

중요한 건 시점이다. 번호를 매기는 건 컴파일 때지만, **번호의 정체를 푸는 건 실행 중 그 명령을 처음 만났을 때**다. 미리 다 풀어두지 않고 필요할 때 푸는 지연(lazy) 방식이라, 한 번 해석된 항목은 주소가 캐싱되어 다음부터는 곧바로 간다.

## 5. 인터프리터와 JIT — 통역사와 번역가의 분업

이제 마지막 단계, 바이트코드를 CPU가 알아듣는 진짜 기계어로 만드는 일이다. JVM은 두 방식을 **섞어서** 쓴다.

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| 인터프리터 | 바이트코드를 한 줄씩 읽으며 즉석 실행 | 시작이 빠름 | 매번 통역하므로 반복 실행이 느림 |
| JIT 컴파일러 | 자주 실행되는 코드를 통째로 기계어로 번역 후 캐싱 | 번역 후엔 기계어 속도 | 번역 자체에 시간이 듦 |

전략은 이렇다. 프로그램은 일단 **인터프리터**로 시작한다(즉시 실행 가능). 실행하면서 JVM이 몰래 통계를 낸다 — "이 메서드, 벌써 수천 번째 호출인데?" 이렇게 뜨겁게 달아오른 코드(hot spot)를 감지하면 **JIT(Just-In-Time) 컴파일러**가 그 부분만 해당 CPU의 기계어로 통번역해서 저장해 둔다. 이후 그 코드는 통역 없이 기계어로 직행한다.

그래서 자바 프로그램에는 독특한 성질이 있다. **돌수록 빨라진다.** 시작 직후엔 통역 모드라 다소 느리다가, 자주 쓰는 경로가 기계어로 굳어지면서 속도가 오른다. 참고로 오라클 JVM의 이름이 "HotSpot"인 것도 이 뜨거운 코드 감지 기법에서 왔다.

실행하면서 JVM이 호출 횟수를 집계하는데 일정 기준 이상 반복 실행되는 코드를 감지하면 — 이런 지점을 실행 빈도를 온도에 빗대 **핫스팟(hot spot)**이라고 부른다 — JIT 컴파일러가 그 부분만 해당 CPU의 기계어로 번역해 저장해 두기 때문이다. 

그리고 이 단계가 "어디서든 실행"의 마지막 퍼즐이다. 어떤 기계어로 번역할지는 **그 기계에 설치된 JVM**이 안다. .class 파일은 어디서나 동일하고, 맞춤 번역은 각 플랫폼의 JVM이 담당한다. 맞춤의 기준은 운영체제라기보다 정확히는 **CPU 아키텍처(x86, ARM 등)**다.

## 6. 전체 여정을 한 문장씩으로

```
① javac가 .java를 컴파일
   → .class 생성 (번호로 지시하는 바이트코드 본문 + 번호의 정체가 적힌 상수풀 색인)

② 실행 시 클래스 로더가 이 세트 전체를 메타스페이스에 로드
   → 이때부터 "런타임 상수풀". 클래스마다 하나씩

③ JVM이 바이트코드 실행 중 #번호를 만나면
   → 그 클래스의 런타임 상수풀에서 정체를 해석(resolve), 결과는 캐싱

④ 인터프리터가 즉석 실행하고,
   → 뜨거운 코드는 JIT가 그 기계 CPU의 기계어로 번역해 재사용
```

## 마무리 — 구조를 알면 코드가 달리 보인다

이 파이프라인을 알고 나면 평소 겪던 현상들의 이유가 보이기 시작한다. 왜 자바는 실행 파일이 아니라 JVM 설치를 요구하는지(②의 번역기가 필요해서), 왜 서버 애플리케이션이 기동 직후보다 워밍업 후에 빨라지는지(④의 JIT), 왜 `NoClassDefFoundError`가 컴파일이 아니라 실행 중에 터지는지(③의 지연 로딩·해석). 문법만 알 때는 "원래 그런가 보다" 하고 넘기던 것들이 전부 구조의 필연적인 결과였던 셈이다.

이런 구조적 이해가 코드 작성에 직접 주는 이득도 있다. 문자열 리터럴이 왜 공유되는지 알면 `==` 비교의 함정이 안 무섭고, 클래스가 언제 로드되는지 알면 static 초기화 타이밍 버그를 예측할 수 있다. 시리즈 내내 메모리 그림을 먼저 그리고 문법을 얹어온 이유이기도 하다.

다음 글에서는 예고했던 대로 **메서드와 static**을 다룬다. static 필드가 왜 힙의 java.lang.Class 객체 이야기와 연결되는지, 이번 글의 클래스 로딩 타이밍과 static 초기화가 어떻게 맞물리는지 — 오늘 그린 파이프라인 위에서 풀어보겠다.
