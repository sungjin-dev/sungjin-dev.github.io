데이터베이스를 공부하다 보면 가장 처음 마주하는 큰 산이 바로 두 개의 테이블을 합치는 JOIN(조인)이다. 오늘은 시뮬레이터를 돌려보며 깨달은 INNER JOIN과 LEFT JOIN의 차이, 그리고 데이터의 결측치를 다루는 방식을 정리해보자.

## 1. 깐깐한 교집합, INNER JOIN

<img width="1378" height="1260" alt="image" src="https://github.com/user-attachments/assets/a89f1d75-1f7b-401a-929d-3df899887f5c" />

예시에서 Payments.good_id 와 Goods.good_id 가 같다고 명시했으므로, 두 테이블의 good_id 값 이 동일한 레코드만 결과 쿼리에 포함된다.

쉽게 말하면 두 테이블을 가로로 이어 붙일 때, 연결고리(조건)가 양쪽 테이블에 모두 존재하는 행(Row)만 살아남는다.

 데이터베이스 설계에서 **'참조 무결성(Referential Integrity)'**이 완벽하게 지켜진다면, INNER JOIN을 했을 때 데이터가 누락될 걱정은 하지 않아도 된다.
 

## 2. 관대한 합집합, LEFT OR RIGHT JOIN

기준이 되는 왼쪽 혹은 오른쪽 테이블의 모든 행을 무조건 유지한다. 왼쪽 혹은 오른쪽 테이블에 매칭되는 데이터가 있다면 이어 붙이고, 만약 매칭되는 짝꿍이 없다면 없다면 NULL로 채운다. 

일반적인 엑셀 파일이었다면 짝꿍이 없는 줄에서 데이터가 한쪽으로 쏠리며 표 전체가 망가졌을지도 모른다. 하지만 관계형 데이터베이스 프로그램(RDBMS)은 똑똑하게도 짝이 없는 빈칸에는 NULL이라는 특수한 값을 채워 넣어서, 절대 데이터가 옆으로 밀리는 대참사가 일어나지 않도록 줄을 맞춰준다.

```
SELECT *
FROM Users
LEFT JOIN Orders
ON Users.회원ID = Orders.회원ID;
```
<img width="1024" height="936" alt="image" src="https://github.com/user-attachments/assets/f21a2116-f9f6-4830-bf30-f92421d33ed6" />

LEFT JOIN은 왼쪽 테이블(Payments)의 모든 행을 가져오고, 오른쪽 테이블(Goods)에서는 일치하는 데이터만 가져오면 끝이다. 

한가지 헷갈릴 수 있는 부분은 철수는 주문을 2번 했기 때문에 각각 행을 따로 적어줘야한다. 


## 3. 0 vs 공백 vs NULL 

<img width="1024" height="936" alt="image" src="https://github.com/user-attachments/assets/888644ae-6f80-47d5-aab1-eb34cc900ffc" />


데이터베이스에서 NULL은 0이나 띄어쓰기(공백)와 완전히 다른 개념이다.

0 (숫자 데이터): 시험을 치르고 0점을 받은 상태 (명확한 정보)

공백 (문자 데이터): 주관식 답안지를 지우개로 깨끗하게 지워서 백지로 낸 상태 (의도된 빈칸)

NULL (진짜 무(無)): 시험장에 나타나지 않은 결시생. 점수를 알 수 없는 상태 (Unknown)

이 '알 수 없음(Unknown)'의 특성 때문에 NULL은 아래와 같은 독특한 규칙을 가진다.

연산을 집어삼킨다: 100 + NULL = NULL (모르는 값에 100을 더해도 모르는 값일 뿐이다)

= 기호로 찾을 수 없다: WHERE 점수 = NULL (X)  WHERE 점수 IS NULL (O)


## 4. 핵심 요약

INNER JOIN: 양쪽 다 있는 것만 (교집합)

LEFT OR RIGHT JOIN: 왼쪽은 다 살리고, 오른쪽 없으면 NULL 채우기!

NULL의 의미: 0도, 공백도 아닌 완전한 '알 수 없음(Unknown)'
