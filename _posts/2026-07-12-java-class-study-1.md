# 자바 클래스와 객체, 그리고 생성자 

지난 글에서 참조 타입을 정리하며 "클래스는 별도의 챕터가 필요한 큰 주제"라고 미뤄뒀다. 이제 그 본체를 다룰 차례다. 이 글에서는 클래스가 무엇인지, 객체가 어떻게 만들어지는지, 그리고 그 탄생 순간을 담당하는 생성자까지를 정리한다. 지난 글의 메모리 그림(스택·힙·메타스페이스)이 그대로 이어지니, 그 그림을 머리에 띄워놓고 읽으면 두 배로 잘 읽힐 것이다.

---

## 1. 객체 지향의 출발 — 프로그램을 "부품"으로 조립한다

자바는 객체 지향 언어다. 거창하게 들리지만 발상 자체는 단순하다. **현실 세계가 사물(객체)들의 상호작용으로 굴러가듯, 프로그램도 객체라는 부품들을 만들고 조립해서 완성하자**는 것이다.

그렇다면 객체(Object)란 무엇인가. 자바에서 객체는 딱 두 가지로 구성된다.

| 구성 요소 | 의미 | 예: "블로그 회원" 객체라면 |
|-----------|------|---------------------------|
| 필드 (field) | 객체가 가진 **데이터** (속성) | 이름, 이메일, 가입일 |
| 메서드 (method) | 객체가 할 수 있는 **동작** (기능) | 글쓰기, 댓글 달기, 탈퇴하기 |

"데이터와 그 데이터를 다루는 동작을 한 덩어리로 묶은 것" — 이게 객체의 정의다. 데이터 따로, 함수 따로 흩어져 있던 것을 하나의 부품으로 포장했다고 생각하면 된다.

---

## 2. 클래스와 객체 — 설계도와 실물

여기서 핵심 관계가 나온다. **클래스(Class)는 객체를 만들기 위한 설계도이고, 객체는 그 설계도로 찍어낸 실물**이다.

이 비유가 그냥 비유가 아닌 이유가 있다. 지난 글에서 메타스페이스를 "순수 설계도(클래스 구조) 보관소"라고 했던 걸 기억할 것이다. 그 설계도가 바로 클래스다. 실제 메모리에서도 정확히 이 관계로 움직인다.

```
클래스 (설계도)          객체 (실물)
────────────────        ─────────────────
메타스페이스에 1부만       힙에 몇 개든 생성 가능
로드되어 보관됨           (설계도 1부 → 건물 여러 채)
```

용어 하나 정리하고 가자. 클래스로부터 만들어진 객체를 그 클래스의 **인스턴스(instance)**라고 부른다. "객체"와 "인스턴스"는 거의 같은 말인데, 인스턴스는 "어느 클래스 출신인지"를 강조할 때 쓴다. "이 객체는 Member 클래스의 인스턴스다"처럼.

설계도와 실물을 구분하는 감각이 중요한 이유는, **설계도에는 아무리 그림을 그려도 실제 데이터가 생기지 않기 때문**이다. 아파트 설계도에 "안방"을 그렸다고 어딘가에 안방이 생기는 게 아니듯, 클래스를 선언했다고 메모리에 필드 값이 생기지 않는다. 실물은 반드시 `new`로 지어야 한다.

---

## 3. 클래스 선언 — 설계도 그리기

클래스는 세 가지 요소로 구성된다. 블로그 회원을 표현하는 클래스를 직접 그려보자.

```java
public class Member {
    // ① 필드: 객체의 데이터
    String name;
    String email;
    int postCount;

    // ② 생성자: 객체가 생성될 때 실행되는 초기화 담당 (4절에서 자세히)
    Member(String name, String email) {
        this.name = name;
        this.email = email;
    }

    // ③ 메서드: 객체의 동작
    void writePost() {
        postCount = postCount + 1;
        System.out.println(name + "님이 글을 작성했습니다. (총 " + postCount + "개)");
    }
}
```

세 요소의 역할을 정리하면:

| 요소 | 역할 | 특징 |
|------|------|------|
| 필드 | 데이터 저장 | 객체마다 **각자의 값**을 가진다 |
| 생성자 | 객체 생성 시점의 초기화 | 리턴 타입이 없고, 이름이 클래스와 같다 |
| 메서드 | 동작 정의 | 필드를 읽고 쓰며 일한다 |

파일 규칙도 짚고 가자. 소스 파일 이름은 public 클래스 이름과 같아야 한다(`Member.java`). 컴파일하면 클래스마다 `.class` 파일이 생기고, 프로그램 실행 시 이 설계도가 메타스페이스에 로드된다. 지난 글의 그림이 여기서 다시 등장하는 것이다.

## 4. new — 설계도에서 실물을 짓는 순간

설계도가 준비됐으니 실물을 지어보자. 객체 생성은 `new` 연산자가 담당한다.

```java
public class MemberExample {
    public static void main(String[] args) {
        Member m1 = new Member("SungJin", "sungjin@example.com");
        Member m2 = new Member("Mina", "mina@example.com");

        m1.writePost();   // SungJin님이 글을 작성했습니다. (총 1개)
        m1.writePost();   // SungJin님이 글을 작성했습니다. (총 2개)
        m2.writePost();   // Mina님이 글을 작성했습니다. (총 1개)
    }
}
```

`new Member(...)` 한 줄에서 벌어지는 일은, 지난 글에서 배열을 만들 때 봤던 3단계 그대로다.

1. **힙에 객체 생성** — Member 설계도대로 필드(name, email, postCount)를 가진 실물이 힙에 지어진다
2. **생성자 실행** — 갓 지어진 객체의 필드에 초기값이 채워진다
3. **번지 리턴** — 완성된 객체의 번지가 스택의 변수 `m1`에 저장된다

```
   스택 영역              힙 영역
┌───────────┐    ┌─────────────────────────────────┐
│  m1 ●─────┼───→│ Member 객체 #1                   │
│           │    │  name: "SungJin"                │
│           │    │  email: "sungjin@example.com"   │
│           │    │  postCount: 2                   │
│  m2 ●─────┼─┐  ├─────────────────────────────────┤
└───────────┘ └─→│ Member 객체 #2                   │
                 │  name: "Mina"                   │
                 │  email: "mina@example.com"      │
                 │  postCount: 1                   │
                 └─────────────────────────────────┘
```

출력 결과가 증명하는 핵심이 하나 있다. m1이 글을 두 번 써도 m2의 postCount는 1이다. **같은 설계도에서 나왔어도, 각 인스턴스는 자기만의 필드 값을 따로 가진다.** 설계도는 하나지만 각 집의 살림살이는 별개인 것과 같다. 그리고 `m1`은 객체 그 자체가 아니라 객체를 가리키는 참조 변수라는 것 — 지난 글의 규칙이 여기서도 그대로 적용된다.

필드에 값을 안 넣으면 어떻게 될까. 힙에 생성되는 순간 필드는 **타입별 기본값**으로 자동 초기화된다. 숫자는 0, boolean은 false, 참조 타입은 null. 배열의 기본값 규칙과 완전히 같다. 위 예제에서 postCount를 생성자에서 건드리지 않았는데도 0에서 시작한 이유다.

## 5. 생성자 — 객체의 탄생을 책임지는 특별한 블록

생성자(constructor)는 `new` 뒤에 붙어서, **객체가 생성되는 바로 그 순간에 딱 한 번 실행되는 초기화 전담 블록**이다. 생김새는 메서드와 비슷하지만 두 가지가 다르다.

- 이름이 **반드시 클래스 이름과 같다**
- **리턴 타입 선언이 없다** (void도 쓰지 않는다)

```java
public class Member {
    String name;

    Member(String name) {   // 리턴 타입 없음, 이름은 클래스와 동일
        this.name = name;
    }
}
```

### 기본 생성자 — 안 만들면 컴파일러가 만들어준다, 단...

생성자를 하나도 선언하지 않으면, 컴파일러가 내용이 빈 **기본 생성자(default constructor)**를 자동으로 넣어준다. 그래서 생성자 없는 클래스도 `new Member()`가 가능했던 것이다.

그런데 여기에 입문자를 반드시 한 번 잡아먹는 함정이 있다. **생성자를 하나라도 직접 선언하는 순간, 컴파일러는 기본 생성자를 만들어주지 않는다.**

```java
public class Member {
    String name;

    Member(String name) {    // 생성자를 직접 선언했으므로
        this.name = name;
    }
}

Member m = new Member();     // 컴파일 에러! 매개변수 없는 생성자가 이제 없다
```

"어제까지 되던 `new Member()`가 갑자기 안 돼요"의 90%가 이 경우다. 매개변수 없는 생성도 계속 지원하고 싶다면 기본 생성자를 직접 추가해야 한다.

### 생성자 오버로딩 — 짓는 방법을 여러 개 제공하기

객체를 생성하는 방식이 하나일 필요는 없다. 매개변수의 타입·개수·순서를 다르게 해서 생성자를 여러 개 두는 것을 **생성자 오버로딩(overloading)**이라고 한다.

```java
public class Member {
    String name;
    String email;
    String role;

    Member(String name) {
        this.name = name;
        this.email = "미등록";
        this.role = "일반회원";
    }

    Member(String name, String email) {
        this.name = name;
        this.email = email;
        this.role = "일반회원";
    }

    Member(String name, String email, String role) {
        this.name = name;
        this.email = email;
        this.role = role;
    }
}
```

```java
Member m1 = new Member("SungJin");
Member m2 = new Member("SungJin", "sungjin@example.com");
Member m3 = new Member("SungJin", "sungjin@example.com", "관리자");
```

호출할 때 넘긴 매개변수의 형태를 보고 자바가 알아서 맞는 생성자를 골라 실행한다. 이용자 입장에서는 "이름만 알아도, 전부 알아도" 객체를 만들 수 있으니 편해진다.

### this() — 생성자 간의 중복 제거

그런데 위 코드를 보면 세 생성자가 거의 같은 대입문을 반복하고 있다. 이럴 때 `this()`를 쓰면 **한 생성자가 같은 클래스의 다른 생성자를 호출**할 수 있다.

```java
public class Member {
    String name;
    String email;
    String role;

    Member(String name) {
        this(name, "미등록", "일반회원");    // 아래 3-매개변수 생성자에게 위임
    }

    Member(String name, String email) {
        this(name, email, "일반회원");       // 역시 위임
    }

    Member(String name, String email, String role) {
        this.name = name;                    // 실제 초기화는 여기 한 곳에서만
        this.email = email;
        this.role = role;
    }
}
```

초기화 로직을 "가장 완전한 생성자" 한 곳에 모으고, 나머지는 기본값을 채워 그쪽으로 넘기는 패턴이다. 실무에서 매우 흔한 스타일이니 눈에 익혀두자. 규칙 하나: **`this()` 호출은 생성자의 첫 줄에서만 가능**하다.

## 6. this — "지금 이 객체 자신"을 가리키는 참조

생성자 코드에 계속 등장한 `this`를 정리할 차례다. `this`는 **이 코드를 실행하고 있는 객체 자기 자신의 번지를 담은 참조**다.

왜 필요할까. 생성자나 메서드의 매개변수 이름이 필드 이름과 같아지는 순간, 이름이 충돌하기 때문이다.

```java
Member(String name) {
    this.name = name;
    // this.name → 힙에 있는 이 객체의 필드
    //      name → 방금 전달받은 매개변수 (스택의 프레임에 있음)
}
```

지난 글의 메모리 그림으로 보면 명확하다. 매개변수 `name`은 생성자가 실행되는 동안 **스택 프레임**에 사는 지역 변수고, `this.name`은 **힙**에 있는 객체의 필드다. `this`는 "힙에 있는 나 자신"을 콕 집어 가리키는 손가락인 셈이다.

이름이 안 겹치면 `this`를 생략해도 되지만, 필드를 다룬다는 사실이 코드에서 명확해지므로 겹치지 않아도 붙이는 습관을 들이는 걸 추천한다.

## 7. 입문자가 자주 하는 실수 2가지

**① this를 빼먹어서 필드가 초기화되지 않는 경우**

```java
Member(String name) {
    name = name;    // 매개변수를 자기 자신에 대입하는 무의미한 코드!
}

Member m = new Member("SungJin");
System.out.println(m.name);   // null → 필드는 손도 안 댄 상태
```

에러가 나지 않아서 더 무섭다. 컴파일도 실행도 멀쩡한데 필드만 조용히 null로 남는다. 이름이 겹칠 때는 반드시 `this.name = name`이어야 한다.

**② 생성자에 리턴 타입을 붙여버리는 경우**

```java
public class Member {
    void Member(String name) {   // 리턴 타입이 붙는 순간 생성자가 아니라
        ...                       // "Member라는 이름의 평범한 메서드"가 된다!
    }
}
```

이것도 에러가 안 난다. 문법상 유효한 메서드이기 때문이다. 하지만 `new Member("SungJin")`을 해도 이 코드는 실행되지 않는다(생성자가 아니니까). 생성자를 만들었는데 초기화가 안 된다면 리턴 타입이 붙어 있지 않은지부터 확인하자.

## 마무리 — 한 장 요약

| 개념 | 핵심 한 줄 |
|------|-----------|
| 객체 | 필드(데이터) + 메서드(동작)를 묶은 프로그램의 부품 |
| 클래스 | 객체의 설계도. 메타스페이스에 1부 로드된다 |
| 인스턴스 | 설계도로 힙에 찍어낸 실물. 각자 자기 필드 값을 가진다 |
| new | 힙에 생성 → 생성자 실행 → 번지 리턴의 3단계 |
| 생성자 | 생성 순간 1회 실행되는 초기화 블록. 이름 = 클래스명, 리턴 타입 없음 |
| 기본 생성자 | 생성자가 하나라도 있으면 **자동 생성되지 않는다** |
| 오버로딩 | 매개변수 구성을 달리해 생성 방법을 여러 개 제공 |
| this | 객체 자신의 참조. 필드와 매개변수의 이름 충돌을 해결 |
| this() | 다른 생성자 호출로 초기화 중복 제거. 첫 줄에서만 가능 |

정리하면, 클래스는 설계도이고 객체는 `new`가 힙에 지어 올린 실물이며, 생성자는 그 준공 순간의 초기화 담당이다. 지난 글에서 배운 "스택의 변수가 힙의 객체를 가리킨다"는 그림에, 이제 "그 객체는 어떤 설계도에서 어떻게 태어나는가"가 채워진 셈이다.

다음 글에서는 객체의 동작을 담당하는 **메서드**를 파고든다. 매개변수와 리턴값, 메서드 오버로딩, 그리고 인스턴스 멤버와 정적(static) 멤버의 차이 — static이 왜 힙의 java.lang.Class 객체 이야기와 연결되는지도 그때 밝혀진다.
