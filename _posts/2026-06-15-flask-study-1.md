우선 주체별로 구분해서 공부해보자. 각각의 역할을 명확하게 구분하고 거시적인 흐름을 파악하는 것이 좋다. 

먼저

1. `클라이언트 (Client)`는 서버에 요청하는 고객이다. 

비유적으로 표현하자면 식당의 손님이다.

"무언가를 요청(Request)하는 프로그램이나 기기"를 말하는데 때로는 웹서버 또한 클라이언트가 될 수 있다. 

우리가 네이버에 들어가서 날씨 정보를 요청했다면 네이버 또한 기상청이라는 또다른 서버에게 정보를 요청하는 클라이언트가 된다.  

우리가 매일 사용하는 웹 브라우저(크롬, 사파리)나 스마트폰 앱도 클라이언트고 게임 클라이언트도 마찬가지이다. 

"이 화면 보여줘!", "게임하게 로그인시켜 줘!" 등등 서버에게 주문하듯 요구(Request)하는 것이다. 

다음으로 `서버 (Server)`는 주방이라고 보면 된다. 

정말 쉽게 표현하면 고객의 주문을 받고 처리해서 결과물을 내어주는 거다. 

네이버, 구글  상시 클라이언트의 요청을 기다리고 있는데 무수히 많은 요청들을 처리하고 그 요청한 화면을 보여주면서 응답(Response)하게 된다. 


2. 서버(식당)에 찾아가는 방법

우리도 주문을 하려면 식당에 먼저 가야한다. 

맛집 위치를 정확하게 찾기 위해 GPS를 활용하기도 하는데 

이처럼 서버에 도달하기 위해 ip란 녀석이 굉장한 역할을 담당하고 있다. 


3 IP 주소 (IP Address): 건물의 도로명 주소

인터넷에 연결된 모든 전자기기는 ip 주소를 지니고 있다. (예: 142.250.206.110) 
(참고로 총 4바이트 중 1바이트(1칸)에 들어갈 수 있는 숫자의 가짓수가 2^8(256개)라서 각각의 자리는 0~255까지의 수가 자리한다)

클라이언트가 특정 서버(맛집)를 찾아가려면 무조건 이 IP 주소를 알아야하는데 

여기서 `포트 (Port)`가 바로 건물 안의 호수 또는 식당 층, 위치이다.  

port가 항구라는 의미라는건 모두 알거다. 어찌보면 교통정리를 하는건데 서버마다 전용도로 출입문 정도를 달아놓은거라고 이해하면 편하다. 

즉 IP 주소로 서버 건물 앞까지 도착했다면, "식당 안내점원을 만나기 위해서 몇 번 문으로 들어가야할지 정해야 하는데 이것이 포트다.  

더 생각해보기 - 우리가 구글 네이버같은 서버에 어떻게 찾아갈 수 있는걸끼?

전 세계 수십억 개의 웹사이트가 있는데, 단 하나의 DNS 서버가 그 모든 주소를 관리하고 저장할 수는 없다. 
그래서 전 세계 수많은 DNS 서버들이 다단계처럼 역할을 맡고, 서로 스무고개를 하며 진짜 주소를 찾아낸다. 


4. 그러면 클라이언트와 서버가 어떻게 소통하는가?

그 기준은 HTTP (HyperText Transfer Protocol)표준에 따른다. 

클라이언트와 서버가 서로 데이터를 주고받을 때 쓰는 전 세계 공통의 약속(언어)이다. 이건 인터넷을 해본 사람이라면 누구나 접해봤을거다. 

"나는 한국어, 너는 영어 쓰면 서로 못 알아들으니까, 그냥 'HTTP'라는 규칙에 맞춰서 봉투에 주소 적고, 안에 내용물 넣어서 보내자!" 라고 정해둔 거라고 보면 된다. 

클라이언트가 서버에 보낼 편지(HTTP)를 쓸 때, "내가 지금 이 편지를 보내는 목적"을 편지 봉투 겉면에 명확히 적어놓는다. 

 
GET (Read - 읽기): "그냥 보여줘!"

웹 서핑을 할 때 가장 많이 쓴다. 네이버 뉴스를 클릭하거나, 유튜브 영상을 볼 때 브라우저는 서버에게 GET 요청을 보내는데 데이터가 URL 주소창에 훤히 다 보인다. 
공개되어도 상관없는 데이터는 이런식으로 요청하고 주고 받는다. 

POST (Create - 생성/처리): "몰래 전달해줘!"

아이디/비밀번호를 로그인 폼에 입력해서 넘기거나, 공개되면 안 되는 신상정보나 보안이 필요한 정보를 
주소창에 노출하지 않고, 편지 봉투 안쪽(Body)에 꽁꽁 숨겨서 보냅니다.

PUT (Update - 수정): "덮어씌워 줘!"

내 프로필 사진을 새로운 사진으로 바꾸거나, 작성해 둔 내 글의 내용을 수정할 때 사용한다. 

DELETE (Delete - 삭제): "없애줘!"

내가 쓴 댓글을 삭제하거나, 회원 탈퇴를 할 때 서버에게 보내는 명령입니다.

아름답게도 데이터 흐름이라고 볼 수 있는 CRUD(생성, 읽기, 수정, 삭제)와 완벽하게 짝을 이룬다. 


위 내용을 살펴보면 모두가 엃히고 섥혀서 상호작용하고 연관되어 있음을 알 수 있디. 

지금까지의 과정을 예시로 요약하면,

사용자가 브라우저(클라이언트)에서 유튜브 영상에 댓글을 달고 '등록' 버튼을 누르면
브라우저는 구글의 특정 컴퓨터 주소(IP)를 찾아가 웹 전담 출입문(포트)으로 간다. 
전 세계 공통 양식(HTTP)에 맞춰 편지를 쓰며, 내 댓글 내용을 안전하게 봉투 안에 숨기고 봉투 겉면에는 **"POST(등록해 줘)"**라고 도장을 쾅 찍어서 전달한다. 
그러면 서버는 이를 받아 데이터베이스에 저장한 뒤, "댓글 등록 완료!"라는 화면을 다시 클라이언트에게 보내주게 되는 것이다. 


기초적인 이해는 마쳤으니 실습을 통해 알아보자.

기초학습으로는 파이썬(Python)의 플라스크(Flask)만 돌려봐도 충분하다. 

여기서 Flask란?
Flask(플라스크)는 파이썬(Python)으로 웹 서버나 웹 애플리케이션을 아주 쉽고 빠르게 만들 수 있도록 도와주는

'마이크로 웹 프레임워크(Micro Web Framework)이다.

'마이크로'는 작다 세밀하다 그런 의미가 아니라 "가장 핵심적인 뼈대만 제공한다"는 뜻이며
강력한 라우팅 (Routing) 기능을 가지고 있는 것이 특징이다.

템플릿 엔진 (Jinja2) 내장
웹 서버는 결국 사용자에게 예쁜 HTML 화면을 던져주어야 합니다.

파이썬 코드 안에서 지저분하게 str += '<form>'을 쓰지 않아도 되도록, HTML 파일의 뼈대를 미리 만들어두고 그 빈칸에 파이썬 데이터(이름, 나이 등)를 쏙쏙 채워 넣어서 브라우저로 보내주는 마법 같은 기능을 기본적으로 제공합니다.

간단히 말해, 복잡한 네트워크 통신 규칙이나 서버 구동 원리를 몰라도 파이썬 문법만 알면 누구나 전 세계와 통신하는 웹 서버를 만들 수 있게 해주는 아주 고마운 번역기이자 도구 상자입니다.

라우팅(Routing)이란?

영단어만 보면 '길(Route)을 찾아주는 과정'인데 

웹 개발에서는 사용자가 입력한 주소(URL)를 보고, 서버 안에서 어떤 코드를 실행할지 알맞은 길로 연결해 주는 안내 데스크 역할이라고 보면 된다.

1. 클라이언트의 요청

사용자가 브라우저 주소창에 http://127.0.0.1:5000/login 이라고 입력하고 엔터를 칩니다.
IP(127.0.0.1)와 포트(5000)를 찾아 서버 건물 1층까지는 무사히 도착했습니다. 하지만 건물 안에서 어떤 부서로 가야 할지는 아직 모릅니다.

2. 라우터(Router)의 분류 작업 (길 안내)
서버 건물 1층 안내 데스크 직원(라우터)이 사용자가 들고 온 주소의 끝부분인 /login (경로, Path)을 확인합니다.

그러면 직원은 "어디 보자... /about은 소개팀이고... 아! /login은 3층 보안팀(login 담당 함수)으로 보내면 되는구나!"

3. 목적지 도달 및 처리 (함수 실행)
안내 직원의 지시에 따라 요청은 파이썬 코드 안의 @app.route('/login') 이 붙어있는 함수로 전달됩니다.
그 함수가 실행되면서 "로그인 폼 화면"이나 "로그인 성공 데이터"를 클라이언트에게 다시 반환해 줍니다.

만약 라우팅이 없다면?
라우팅이 없다면 서버는 들어오는 모든 요청에 대해 똑같은 한 가지 화면만 보여주거나, "네가 뭘 원하는지 모르겠어"라며 고장 나버릴 것입니다. 
/news를 치면 뉴스 함수를, /webtoon을 치면 웹툰 함수를 실행하게 해주는 핵심 연결 고리가 바로 라우팅입니다.

사용자가 브라우저 주소창에 /login을 치고 오면 로그인 담당 함수로,
/search를 치고 오면 검색 담당 파이썬 함수로 길을 안내해 줍니다. (@app.route 문법)

HTTP 통신(GET, POST 등)의 목적에 따라 알아서 택배를 분류해 주는 아주 똑똑한 길잡이 역할을 합니다.


 
@app.route('/')   # 메인페이지
def home():       # 서버에서 자동 실행 
    return "<h1>Hello Flask</h1>"
 

1. 공간적 한계 (Storage)

만약 넷플릭스나 멜론 앱이 서버를 쓰지 않고 모든 영화와 음악 데이터를 내 스마트폰에 저장해야 한다면, 
앱 하나 다운로드하는 데 스마트폰 용량이 수백 테라바이트(TB)가 필요할 것입니다. 불가능한 일이죠.

2. 데이터 공유 (Synchronization)

카카오톡 단체방을 생각해 보면 쉽습니다. 내 스마트폰(클라이언트)에만 메시지를 저장한다면 다른 사람들은 내가 쓴 글을 볼 수가 없겠죠. 
모두가 같은 화면을 공유하려면, 데이터는 무조건 가운데에 있는 '중앙 서버'를 거쳐서 뿌려져야만 합니다.

3. 보안상의 문제 (Security)

내 은행 계좌 잔고 데이터가 내 스마트폰(클라이언트) 안에 저장되어 있다면? 
해커나 나쁜 마음을 먹은 사람이 스마트폰의 데이터를 조작해서 잔고를 100억 원으로 바꿔버릴 수 있습니다. 
그래서 잔고(데이터)는 절대 조작할 수 없는 안전한 금고(서버)에 숨겨두고, 
클라이언트는 그저 "내 잔고 얼마야?" 하고 묻기만 해야 합니다.

 "API는 약속이다"
이러한 요청을 보내는 방법을 약속해둔 것이 API이다
클라이언트와 서버는 서로 다른 언어를 쓰고 다른 환경에 있습니다. 그래서 "네가 A라고 요청하면, 나는 B라는 형식으로 데이터를 줄게"라고 미리 정해둔 규칙, 계약서, 혹은 메뉴판이 반드시 필요한데, 그 메뉴판의 정식 명칭이 바로 API(Application Programming Interface)인 것입니다.

지금까지 배우신 Flask 라우팅(@app.route('/login')) 코드가 바로, 프론트엔드 개발자에게 "로그인하고 싶으면 /login 주소로 아이디랑 비밀번호를 POST 방식으로 보내!"라고 약속(API)을 만들어주는 과정이었습니다.
