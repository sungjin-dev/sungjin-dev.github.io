먼저

## 1. 클라이언트 (Client)는 서버에 요청하는 고객이다.

비유적으로 표현하자면 식당의 손님이다.
<br>
"무언가를 요청(Request)하는 프로그램이나 기기"를 말하는데 때로는 웹서버 또한 클라이언트가 될 수 있다. 
우리가 네이버에 들어가서 날씨 정보를 요청했다면 네이버 또한 기상청이라는 또다른 서버에게 정보를 요청하는 클라이언트가 된다.  
우리가 매일 사용하는 웹 브라우저(크롬, 사파리)나 스마트폰 앱도 클라이언트고 게임 클라이언트도 마찬가지이다. 
"이 화면 보여줘!", "게임하게 로그인시켜 줘!" 등등 서버에게 주문하듯 요구(Request)하는 것이다. 
<br><br>
다음으로 **서버 (Server)는 주방**이라고 보면 된다. 
<br><br>
정말 쉽게 표현하면 고객의 주문을 받고 처리해서 결과물을 내어주는 거다. 
네이버, 구글  상시 클라이언트의 요청을 기다리고 있는데 무수히 많은 요청들을 처리하고 그 요청한 화면을 보여주면서 응답(Response)하게 된다. 

## 2. 서버(식당)에 찾아가는 방법

우리도 주문을 하려면 식당에 먼저 가야한다. 
맛집 위치를 정확하게 찾기 위해 GPS를 활용하기도 하는데 
이처럼 서버에 도달하기 위해 ip란 녀석이 굉장한 역할을 담당하고 있다. 

## 3. IP 주소 (IP Address): 건물의 도로명 주소

인터넷에 연결된 모든 전자기기는 `ip 주소`를 지니고 있다. (예: 142.250.206.110) 
(참고 IPv4 기준 총 4바이트 중 1바이트(1칸)에 들어갈 수 있는 숫자의 가짓수가 2^8(256개)라서 각각의 자리는 0~255까지의 수가 자리한다)
<br><br>
클라이언트가 특정 서버(맛집)를 찾아가려면 무조건 이 IP 주소를 알아야하는데 
여기서 `포트 (Port)`가 바로 건물 안의 호수 또는 식당 층, 위치이다. 즉 하나의 컴퓨터(IP) 안에서도 여러 서버 프로그램이 동시에 실행될 수 있기 때문에
포트 번호를 통해 어떤 프로그램에게 요청을 전달할지 구분한다. 
<br><br>
port가 항구라는 의미라는건 모두 알거다. 어찌보면 교통정리를 하는건데 서버마다 전용도로 출입문 정도를 달아놓은거라고 이해하면 편하다. 
즉 IP 주소로 서버 건물 앞까지 도착했다면, "식당 안내점원을 만나기 위해서 몇 번 문으로 들어가야할지 정해야 하는데 이것이 포트다.  
<br><br>
> **더 생각해보기** - 우리가 구글 네이버같은 서버에 어떻게 찾아갈 수 있는걸?
> 전 세계 수십억 개의 웹사이트가 있는데, 단 하나의 `DNS 서버`가 그 모든 주소를 관리하고 저장할 수는 없다. 
> 그래서 전 세계 수많은 `DNS 서버`들이 다단계처럼 역할을 맡고, 서로 스무고개를 하며 진짜 주소를 찾아낸다.
<br><br>
그렇다고 전 세계의 DNS 서버가 아무렇게나 서로 묻는 것은 아니다.
`DNS`는 루트 DNS 서버 → 최상위 도메인(.com, .net 등) 서버 → 권한 DNS 서버(Authoritative DNS) 순으로 체계적인 `계층 구조`를 가지고 있다.

마치 "대한민국 → 대전광역시 → 서구 → 둔산동" 순서로 주소를 찾아가는 것과 비슷하다. 

## 4. 그러면 클라이언트와 서버가 어떻게 소통하는가?

그 기준은 **`HTTP (HyperText Transfer Protocol)`**표준에 따른다. 

클라이언트와 서버가 서로 데이터를 주고받을 때 쓰는 전 세계 공통의 약속(언어)이다. 이건 인터넷을 해본 사람이라면 누구나 접해봤을거다. 
"나는 한국어, 너는 영어 쓰면 서로 못 알아들으니까, 그냥 'HTTP'라는 규칙에 맞춰서 봉투에 주소 적고, 안에 내용물 넣어서 보내자!" 라고 정해둔 거라고 보면 된다. 
클라이언트가 서버에 보낼 편지(HTTP)를 쓸 때, "내가 지금 이 편지를 보내는 목적"을 편지 봉투 겉면에 명확히 적어놓는다. 
<br><br>
* **`GET` (Read - 읽기): "그냥 보여줘!"**
    웹 서핑을 할 때 가장 많이 쓴다. 네이버 뉴스를 클릭하거나, 유튜브 영상을 볼 때 브라우저는 서버에게 GET 요청을 보내는데 주로 URL에 요청 정보가 포함되기 때문에
    공개되어도 상관없는 데이터는 이런식으로 요청하고 주고 받는다. 
<br><br>
* **`POST` (Create - 생성/처리): "몰래 전달해줘!"**
    아이디/비밀번호를 로그인 폼에 입력해서 넘기거나, 공개되면 안 되는 신상정보나 보안이 필요한 정보를 
    주소창에 노출하지 않고, 데이터를 URL이 아닌 요청 `본문(Body)`에 담아 전송한다. 단, POST 자체가 데이터를 암호화하는 것은 아니며 보안은 `HTTPS`가 담당한다.
<br><br>
* **`PUT/PATCH` (Update - 수정): "덮어씌워 줘!"**
    내 프로필 사진을 새로운 사진으로 바꾸거나, 작성해 둔 내 글의 내용을 수정할 때 사용한다. 실무에서는 전체 수정은 `PUT`, 일부 수정은 `PATCH`를 사용하는 경우가 많다.
<br><br>
* **`DELETE` (Delete - 삭제): "없애줘!"**
    내가 쓴 댓글을 삭제하거나, 회원 탈퇴를 할 때 서버에게 보내는 명령이다.
<br><br>
아름답게도 데이터 흐름이라고 볼 수 있는 `CRUD(생성, 읽기, 수정, 삭제)`와 완벽하게 짝을 이룬다. 
<br><br>
위 내용을 살펴보면 모두가 얽히고설켜서 상호작용하고 연관되어 있음을 알 수 있. 
<br><br>
지금까지의 과정을 예시로 요약하면,
사용자가 브라우저(클라이언트)에서 유튜브 영상에 댓글을 달고 '등록' 버튼을 누르면
브라우저는 구글의 특정 컴퓨터 주소(IP)를 찾아가 웹 전담 출입문(포트)으로 간다. 
전 세계 공통 양식`(HTTP)`에 맞춰 편지를 쓰며, 내 댓글 내용을 안전하게 봉투 안에 숨기고 봉투 겉면에는 **"POST(등록해 줘)"**라고 도장을 쾅 찍어서 전달한다. 
그러면 서버는 이를 받아 데이터베이스에 저장한 뒤, "댓글 등록 완료!"라는 화면을 다시 클라이언트에게 보내주게 되는 것이다. 
<br><br>
기초적인 이해는 마쳤으니 실습을 통해 알아보자.
기초학습으로는 파이썬(Python)의 플라스크(Flask)만 돌려봐도 충분하다. 

## 5. 여기서 Flask란?
**Flask(플라스크)**는 파이썬(Python)으로 웹 서버나 웹 애플리케이션을 아주 쉽고 빠르게 만들 수 있도록 도와주는
`마이크로 웹 프레임워크(Micro Web Framework)`이다.
<br><br>
'마이크로'는 작다 세밀하다 그런 의미가 아니라 "가장 핵심적인 뼈대만 제공한다"는 뜻이며
강력한 라우팅 (Routing) 기능을 가지고 있는 것이 특징이다.

**템플릿 엔진 (Jinja2) 내장**
웹 서버는 결국 사용자에게 예쁜 HTML 화면을 보여줘야 하는데 이 엔진을 이용하면 HTML 파일에 파이썬 데이터를 쉽게 삽입할 수 있으며 동적인 웹 페이지를 효율적으로 생성할 수 있다.
<br><br>
간단히 말해, 복잡한 네트워크 통신 규칙이나 서버 구동 원리를 몰라도 파이썬 문법만 알면 누구나 전 세계와 통신하는 웹 서버를 만들 수 있게 해주는 아주 고마운 번역기이자 도구 상자이다.
<br><br>
**라우팅(Routing)이란?**
영단어만 보면 '길(Route)을 찾아주는 과정'인데 
웹 개발에서는 사용자가 입력한 주소(URL)를 보고, 서버 안에서 어떤 코드를 실행할지 알맞은 길로 연결해 주는 안내 데스크 역할이라고 보면 된다.
<br><br>
1. **클라이언트의 요청:** 사용자가 브라우저 주소창에 [http://127.0.0.1:5000/login] 이라고 입력하고 엔터를 친다고 가정하면 IP(127.0.0.1)와 포트(5000)를 찾아 서버 건물 1층까지는 무사히 도착한거다. 하지만 건물 안에서 어떤 부서로 가야 할지는 아직 모르는 상황.
<br><br>
3. **라우터(Router)의 분류 작업 (길 안내):** 서버 건물 1층 안내 데스크 직원(라우터)이 사용자가 들고 온 주소의 끝부분인 /login (경로, Path)을 확인하고  /about은 소개팀으로 /login은 3층 보안팀으로 보낸다. 
<br><br>
5. **목적지 도달 및 처리 (함수 실행):** 안내 직원의 지시에 따라 요청은 파이썬 코드 안의 @app.route('/login') 이 붙어있는 함수로 전달된다 그 함수가 실행되면서 "로그인 폼 화면"이나 "로그인 성공 데이터"를 클라이언트에게 다시 반환해준다. 
<br><br>
만약 라우팅이 없다면 서버는 들어오는 모든 요청에 대해 똑같은 한 가지 화면만 보여주거나, "네가 뭘 원하는지 모르겠어"라며 고장 나버린다. 
`/news`를 치면 뉴스 함수를, `/webtoon`을 치면 웹툰 함수를 실행하게 해주는 핵심 연결 고리가 바로 `라우팅`인 것이다. 
<br><br>
사용자가 브라우저 주소창에 `/login`을 치고 오면 로그인 담당 함수로,
`/search`를 치고 오면 검색 담당 파이썬 함수로 길을 안내해 줍니다. (@app.route가 길을 뚫어주는 것!)
`HTTP 통신(GET, POST 등)`의 목적에 따라 알아서 택배를 분류해 주는 아주 똑똑한 길잡이 역할을 담당한다.

<참고> 플라스크(Python Server) 예시
<br><br>
```python
@app.route('/')   # 메인페이지
def home():       # 서버에서 자동 실행 
    return "<h1>Hello Flask</h1>"
```
```python
@app.route('/login')
def login():
    return "로그인 페이지"

@app.route('/about')
def about():
    return "소개 페이지"
```

> 이렇게 주소별로 함수가 연결되는걸 확인할 수 있다. 
<br><br>

home()
      │
      │ HTML 생성
      ▼
[브라우저]
```

```
